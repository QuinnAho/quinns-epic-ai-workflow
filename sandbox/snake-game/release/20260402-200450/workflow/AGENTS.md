# AGENTS.md - Codex Runbook

This file is read by OpenAI Codex at the start of each autonomous session. It contains the operating rules, the current task queue, and the completion signals for this repo.

## Project Constitution

See `PROJECT.md` for the full project constitution for this Codex-first autonomous web-game workflow.

### DO
- Follow the linked spec files and the game's sandbox brief exactly
- Prefer the smallest playable increment over broad speculative rewrites
- Keep the browser artifact runnable, or restore it before finishing
- Extend the game's sandbox test harness as pure logic and smoke checks become available
- Run a self-review pass with `code_reviewer` and `spec_validator` before marking non-trivial tasks complete
- Prepare today's code for clearly planned follow-on systems when the spec calls for them, without speculative overbuilding
- Update `STATUS.md` after each meaningful task with artifact status and blocker notes
- Run mechanical quality gates before committing
- Use GPT-5.4-mini for simple edits, test adjustments, and cleanup when it saves rate limits

### DO NOT
- Wait for a separate review step; perform the reviewer pass yourself and leave blockers for the human only when needed
- Hide broken behavior behind vague TODOs when a concrete fix or failure note is possible
- Break the current playable artifact without replacing it with a better one
- Add heavyweight dependencies unless they clearly improve the game-generation workflow
- Skip tests or smoke checks when a useful check is practical
- Overbuild future systems that are not yet justified by the current spec

## Completion Signals

- `TASK_COMPLETE` - Current task finished, move to the next task
- `BLOCKED` - Cannot proceed without human input, missing assets, or environment changes
- `ALL_TASKS_DONE` - All tasks in the current queue are complete
- `RATE_LIMITED` - Hit rate limits, pause and retry later

## Task Execution Protocol

For each task:

1. Read the linked spec, the game's `sandbox/<game-slug>/idea.txt` brief if it exists, and the current implementation state
2. If the spec file does not exist yet and the task is to create it, create it first
3. Start from the existing `sandbox/<game-slug>/tests/` harness when it exists and add failing tests when practical for the task
4. If tests are not practical because the task is browser-visual or artifact-oriented, add the smallest useful smoke check or document the limitation
5. Implement only the minimum change needed to move the game toward a playable state
6. When the spec names likely follow-on systems, leave light-weight seams or shared data structures that make those later tasks easier
7. Run a self-review pass with `code_reviewer` and `spec_validator` for non-trivial work and fix material findings
8. Run the relevant checks and `./scripts/quality-gate.sh`
9. Update `STATUS.md` with artifact status, failures found, review notes, and blocker notes
10. If the task passes, commit with a descriptive message
11. If blocked, document the blocker clearly and continue to the next thin task

## Blocker Handling

If you encounter a blocker:

1. Do not spend more than 15 minutes on a single dead end
2. Document the blocker in `STATUS.md`:
   - What you tried
   - The concrete error or failure mode
   - What input, asset, or decision is missing
3. Mark the task as `BLOCKED`
4. Move to the next task if possible
5. Leave behind the next useful follow-up task for the human or the next Codex session

## Rate Limit Management

- Track remaining requests via response headers when available
- When below 20% capacity:
  - Switch to GPT-5.4-mini for simple tasks
  - Prefer smaller cleanup steps over broad rewrites
  - Pause if needed instead of spending the last budget on low-signal work

## Artifact Expectations

- Every useful session should leave behind the current browser entry file under `sandbox/<game-slug>/` or record the exact path if a nested layout is used
- Record how to launch the current artifact in `STATUS.md`
- For gameplay tasks, note whether the artifact is better, worse, or unchanged after the run
- Treat collision, movement, minimap accuracy, AI navigation, and delta-time behavior as first-class quality signals

## Subagent Guidance

When a task has clean separable parts, prefer Codex subagents over doing everything in one thread.

- Use `spec_analyst` before implementing a weak or ambiguous spec
- Use `spec_architect` for decomposition and system design
- Use `spec_developer` for bounded implementation work
- Use `spec_tester` for tests or smoke checks that can run in parallel
- Use `spec_validator` to confirm the result matches the spec
- Use `code_reviewer` for read-only review and regression detection
- Use `workflow_integrator` when changing the workflow itself

Do not spawn subagents for trivial work. Use them when parallelism or role separation materially improves the result.

## Notes For Human

Update the task queue before each run. Keep tasks specific, executable, and tied to the current playable artifact.

- Link to spec files or output files explicitly
- State browser-visible acceptance criteria, not just code changes
- Prefer repair loops over giant greenfield tasks
- Queue the next 1-3 tasks only

The more concrete the queue is, the more reliably Codex can move the game forward without supervision.
